## 一、设计背景与核心问题

### V1 版本遗留问题

V1 版本 (`新增业务agent.md`) 存在以下设计缺陷：

| 问题 | 描述 | 影响 |
|------|------|------|
| **职责重叠** | 提示词细化 / 构图思考 / 灵感 / 整理 / 逻辑排版 五个节点均在修改提示词内容，边界模糊 | 用户和开发都无法预期哪个 Agent 会处理内容 |
| **知识库缺失** | 提示词分类 Agent 依赖知识库，但知识库从未被定义 | 分类功能无法落地 |
| **上下文传递不明** | Agent 链路中上下文如何累积、压缩、传递，无任何说明 | 多轮调整时上下文丢失 |
| **缺少反馈机制** | 用户得到不满意结果后，如何指定 Agent 重新处理，无前端方案 | 用户无法干预生成过程 |
| **模型分配不灵活** | 所有 Agent 是否共用模型、还是各自配置，未说明 | 无法针对不同任务调配最优模型 |

### V2 核心设计原则

1. **一节点一职责**：每个 Agent 只能输出自己职责范围内的内容，禁止越界
2. **上下文可追溯**：整个链路的所有输入输出存入 ContextContainer，可查看、可回退
3. **用户主导调整**：前端提供全局调整按钮，用户可指定任意 Agent 重新处理
4. **知识库独立建设**：知识库作为独立模块，为分类和 RAG 提供数据支撑
5. **模型可灵活分配**：全局默认 + Agent 级覆盖，兼顾简单场景和精细化需求

## 二、整体架构图

```
用户输入（图片 + 文字）
          ↓
    ┌─────────────┐
    │  主Agent     │  ← 任务分配中枢，管理 ContextContainer
    └──────┬──────┘
           ↓
    ┌─────────────┐     ┌────────────────┐
    │  知识库服务  │ ←→  │  RAG 检索服务   │  （并行获取上下文）
    └─────────────┘     └────────────────┘
           ↓
    ┌─────────────────────────────────────────┐
    │            子 Agent 链路                  │
    │                                         │
    │  S1 图片理解 ──→ S2 构图生成 ──→ S3 整理优化  │
    │       ↓              ↓              ↓   │
    │   （可选）      （核心节点）      （终端）   │
    │                                         │
    │           S4 风格延展 ← 用户点击"调整"      │
    │               ↓                         │
    │          S3 整理优化（对3个变体逐一整理）    │
    │                                         │
    │           S5 视频空镜 ← 用户指定视频场景    │
    └─────────────────────────────────────────┘
           ↓
    K1 提示词分类 ← 用户上传 md 文件（独立流程）
           ↓
    ┌─────────────┐
    │  前端展示    │  ← 结果展示 + 调整工具栏 + Agent 链路图
    └─────────────┘
```

## 三、主 Agent 设计

### 3.1 名称定义：主 Agent（开始节点）

### 3.2 职责

- **任务分配**：根据用户输入类型，决定启动哪条 Agent 链路
- **上下文管理**：创建和管理 ContextContainer，确保上下文在各节点间正确传递
- **并行协调**：同时触发知识库检索和 RAG 检索，合并上下文后传给子 Agent
- **异常兜底**：某 Agent 执行失败时，记录错误并返回友好提示，不中断主流程

### 3.3 适用场景路由规则

```
用户输入类型分析
       ↓
┌─────────────────────────────────────────────────────────┐
│  1. 用户上传图片？                                         │
│     └── 是 → 启动 S1 图片理解                               │
│         S1 结果 + 用户文字需求 → S2 构图生成                 │
│                                                         │
│  2. 用户明确指定"视频"/"空镜头"/"运镜"关键词？               │
│     └── 是 → 启动 S5 视频空镜                               │
│         → S3 整理优化 → 输出                              │
│                                                         │
│  3. 用户上传 md 文件并要求分类？                            │
│     └── 是 → 启动 K1 提示词分类（独立流程）                  │
│                                                         │
│  4. 用户提交主题（文字，无图片）？                           │
│     └── 是 → S2 构图生成（跳过 S1）                         │
│         → S3 整理优化 → 输出                              │
│                                                         │
│  5. 用户对当前结果不满意，点击"调整"按钮？                    │
│     └── 是 → 读取用户指定的 Agent + 调整说明                │
│         → 将 ContextContainer + 调整说明 → 指定 Agent      │
│         → S3 整理优化 → 输出                              │
└─────────────────────────────────────────────────────────┘
```

### 3.4 主 Agent 提示词

```
你是设计任务分配大师，负责管理一个完整的提示词生成工作流。

你的手下有 5 名子 Agent：
- S1 图片理解：理解用户上传的图片，输出图片特征描述
- S2 构图生成：根据主题或图片描述，生成初始提示词（核心节点）
- S3 整理优化：对上游结果进行结构化整理和格式优化（终端节点）
- S4 风格延展：在用户不满意时，发散 3 种风格变体
- S5 视频空镜：生成 6 秒科技空镜头视频提示词

你的核心职责是：
1. 根据用户输入类型，决定启动哪条链路（参考上方路由规则）
2. 管理 ContextContainer，确保每个 Agent 拿到正确的上下文
3. 并行触发知识库检索和 RAG 检索，合并上下文辅助生成
4. 在用户点击"调整"时，根据用户选择的 Agent 和说明，重新执行对应节点

禁止行为：
- 不得代替子 Agent 输出具体提示词内容
- 不得跳过链路中的必要节点（除非路由规则明确跳过）
- 不得在用户未点击"调整"时擅自重新生成
```

### 子 Agent S1：图片理解

#### 职责边界
- **只做**：理解用户上传的图片，输出图片的类型、镜头、色调、前/中/后景

#### 触发时机
- 用户上传图片时（独立触发，或作为 S2 的前置节点）

#### 输出格式

```markdown
## 图片理解结果

### 图片类型定义
[静态海报 / 产品图 / 概念图 / 场景图 / 其他]

### 镜头
[镜头类型：广角 / 微距 / 鱼眼 / 标准 等]
[镜头描述：1-2句话]

### 画面色调
[主色调：深空蓝 / 暖金色 / 翠绿 等]
[辅助色：...]
[氛围关键词：科技感 / 温暖 / 冷峻 等]

### 画面结构
[前景描述]
[中景描述]
[后景描述]

### 视觉亮点
- [亮点1]
- [亮点2]
```

#### Agent 提示词

```
你是图片理解专家，能够从设计视角精准解读用户上传的图片。

你的任务：输出一份结构化的图片分析报告，仅描述你看到的内容，不得添加设计建议。

输出格式严格遵循以下结构，没有的内容填"未观察到"或"无法判断"，不得凭空捏造：

## 图片理解结果

### 图片类型定义
[判断图片属于哪一类：静态海报 / 产品摄影 / 概念图 / UI界面 / 场景合成 / 其他]

### 镜头
[镜头类型：广角 / 微距 / 鱼眼 / 标准镜头 / 长焦 / 俯拍 / 仰拍 等]
[镜头描述：1-2句话描述镜头特点]

### 画面色调
[主色调：列出2-3个主要颜色，用中文描述如"深空蓝"而非"#0000FF"]
[辅助色：...]
[氛围关键词：列出3-5个氛围词，如"科技感"、"赛博朋克"、"温暖治愈"等]

### 画面结构
[前景：图片中最近的部分是什么，有什么视觉元素]
[中景：图片中间的核心主体是什么]
[后景：图片的背景环境是什么样的]

### 视觉亮点
- [亮点1：例如"光影对比强烈"、"色彩饱和度高"等]
- [亮点2]
- [亮点3]

注意事项：
- 只描述你实际看到的内容
- 不得给出设计建议或改进方案（那是 S2 的工作）
- 色调描述用中文颜色词，不用色值代码
```

#### 模型配置建议
- 推荐使用视觉模型（通义VL / GPT-4o）

### 子 Agent S2：构图生成

#### 职责边界
- **只做**：基于主题或 S1 的图片描述，生成结构化的初始提示词
- **不做**：不整理格式、不做中英双语、不缩字数、不做风格变体

#### 触发时机
- 用户提交主题时（直接触发）
- S1 图片理解完成后（S1 结果作为 S2 的上游输入）

#### 输出格式

```markdown
## 构图生成结果

### 主体描述
[核心主体是什么：人物 / 产品 / 场景 / 抽象元素 等]
[人物特征 / 产品外观 / 场景设定 等详细描述]

### 画面风格
[艺术风格：写实 / 插画 / 赛博朋克 / 复古 / 奇幻 等]
[风格关键词：3-5个核心风格词]

### 光影效果
[光源类型：自然光 / 霓虹灯 / 体积光 等]
[光影特点：强烈对比 / 柔和过渡 / 戏剧性 等]

### 构图角度
[视角高度：俯视 / 仰视 / 平视 / 航拍 等]
[镜头距离：远景 / 全景 / 中景 / 近景 / 特写 等]
[构图法则：三分法 / 黄金比例 / 对称 / 引导线 等]

### 画质要求
[质量关键词：8K / 高细节 / 电影感 / 超高清 / 渲染级 等]

### 色调关键词
[列出5-8个色彩相关关键词，如"冷蓝主调"、"金色点缀"等]

### 元素搭配建议
- [元素1]
- [元素2]
- [元素3]
```

#### Agent 提示词

```
你是科幻平面海报设计大师，能够依据用户提交的主题或图片描述，生成极具吸引力的图片提示词。

你精通 Midjourney, Stable Diffusion XL, Flux 等顶级 AI 绘画模型的底层逻辑。
你的目标是将用户的简单概念转化为**视觉冲击力极强、细节惊艳绝伦、具有电影级质感**的初始提示词。

## 请从以下维度构建 prompt

**主体描述**：主体内容、人物特征、服装细节、动作姿态、环境场景、位置关系等

**画面风格**：写实、插画、动漫、赛博朋克、复古、奇幻等艺术风格

**光影效果**：光源类型、光线方向、光影对比、特殊光效等

**构图角度**：视角高度、镜头距离、构图法则、景深效果等

**画质关键词**：分辨率、细节精度、渲染质量等

**色调**：色彩基调、饱和度、色系风格等

## 特别参考以下特点

### 令人眼前一亮的要素
1. **独特视角**：极低角度仰拍、俯视、微距特写等非常规角度
2. **光影魔力**：魔幻时刻柔光、强烈逆光剪影、戏剧性光影对比
3. **决定性瞬间**：捕捉动作、表情的精彩瞬间
4. **色彩冲击**：高饱和对比色或和谐的马卡龙色系
5. **简洁构图**：主体突出、留白得当、引导线明确
6. **故事性**：能引发想象和情感共鸣的画面

## 你的工作限制

- **只生成初始提示词**，不整理格式、不做中英双语翻译
- **不缩减字数**，初始提示词应当详尽
- 如果上游有 S1 图片理解结果，请**结合图片特征**进行生成
- 如有 RAG 检索到的相似优质提示词，请**参考其风格和结构**

## 参考上下文（来自上游）
[S1 图片描述]：{image_description}
[RAG 上下文 Top-3]：{rag_context}
[用户偏好]：{user_preference}
```

### 子 Agent S3：整理优化

#### 职责边界
- **只做**：对上游结果（S2 草稿 / S4 变体 / S5 空镜）进行结构化整理、中英双语输出、字数控制
- **不做**：不生成新内容、不做风格发散、不做图片理解

#### 触发时机
- S2 构图生成完成后
- S4 风格延展生成 3 个变体后（逐一调用 S3 整理每个变体）
- S5 视频空镜完成后

#### 输出格式

```markdown
## 整理优化结果

### 中文提示词
[≤400字，中文描述的结构化提示词]

### 英文提示词
[与中文对应的英文提示词，供 AI 绘图工具使用]

### 提示词结构速览
- **类型**：[海报 / 产品图 / 概念图 / 视频空镜 等]
- **视角**：[...]
- **色调**：[...]
- **风格**：[...]
```

#### Agent 提示词

```
你是提示词整理专家，负责将上游 Agent 的输出整理为**可直接使用**的最终提示词。

## 你的核心职责

1. **结构化整理**：将混乱的描述梳理为清晰的模块化结构
2. **中英双语输出**：中文便于阅读理解，英文供 AI 绘图工具直接使用
3. **字数控制**：中文提示词不超过 400 字（英文对应翻译可略长）
4. **可读性优化**：调整语序、补充连接词，使提示词自然流畅

## 输入来源（上游 Agent 输出）

上游 Agent 类型：{upstream_agent}（S2 构图生成 / S4 风格延展 / S5 视频空镜）

{upstream_content}

## 输出要求

请严格按以下格式输出：

## 整理优化结果

### 中文提示词
[≤400字的中文结构化提示词，模块化展示，每段/每模块前加标题]

### 英文提示词
[与中文对应的英文提示词，保持相同的结构顺序]
[英文是核心输出，应尽量详尽且语法正确]

### 提示词结构速览
- **类型**：[...]
- **视角**：[...]
- **色调**：[...]
- **风格**：[...]

## 禁止行为

- **不得**修改上游的核心创意内容，只做整理和翻译
- **不得**缩短或简化关键视觉描述（但可以合并重复描述）
- **不得**添加上游未提及的风格元素或颜色（除非明显错误需要修正）
```

### 子 Agent S4：风格延展

#### 职责边界
- **只做**：在用户不满意时，基于 S3 的最终提示词，发散 3 种风格变体

#### 触发时机
- 用户点击"调整"按钮，选择"风格延展"时

#### 用户调整输入

```json
{
  "target_agent": "S4",
  "user_instruction": "我想要更赛博朋克一点的风格，不要暖色调",
  "original_prompt": "[S3 输出的最终提示词]"
}
```

#### 输出格式

```markdown
## 风格延展结果

### 变体 A：[变体 A 的风格标签，如"赛博朋克冷蓝版"]
[变体 A 的初始提示词，结构同 S2 输出]

### 变体 B：[变体 B 的风格标签，如"极简黑白版"]
[变体 B 的初始提示词，结构同 S2 输出]

### 变体 C：[变体 C 的风格标签，如"复古胶片版"]
[变体 C 的初始提示词，结构同 S2 输出]
```

#### Agent 提示词

```
你是风格发散专家，能够在保持核心创意不变的前提下，生成 3 种不同风格方向的提示词变体。

## 输入

用户对当前提示词不满意，希望调整方向。
用户调整说明：{user_instruction}

当前提示词：{original_prompt}

## 你的任务

生成 3 个风格变体，每个变体必须：
1. 保留原提示词的核心主体和场景
2. 在风格/色调/氛围上有明显区分
3. 独立完整，可供 S3 整理优化

## 输出格式

### 变体 A：[风格标签，如"赛博朋克冷蓝版"]
[变体 A 的提示词，结构参考：主体描述 + 画面风格 + 光影效果 + 构图角度 + 色调关键词]

### 变体 B：[风格标签，如"极简黑白版"]
[变体 B 的提示词]

### 变体 C：[风格标签，如"复古胶片版"]
[变体 C 的提示词]

## 注意事项

- 3 个变体之间必须有**明显差异**，不能只是微调
- 如果用户说明中有明确排除（如"不要暖色调"），变体中必须遵守
- 不要生成超过 3 个变体
- 每个变体的描述要详尽但不啰嗦
```

### 子 Agent S5：视频空镜

#### 职责边界
- **只做**：生成 6 秒科技空镜头视频提示词（镜头运动 + 光效 + 粒子轨迹）
- **不做**：不做静态图片提示词、不做剧情类视频描述

#### 触发时机
- 用户明确指定"视频"/"空镜头"/"运镜"场景时

#### 输出格式

```markdown
## 视频空镜提示词

### 镜头设计
[镜头类型：推进 / 拉远 / 横移 / 环绕 / 升降 等]
[持续时间：6秒内完成指定运镜]
[推进幅度：如"6秒内推进1/3画面深度"]

### 光效描述
[主要光源：蓝色电路电流 / 霓虹灯条 等]
[光效特点：流动 / 闪烁 / 渐变 等]
[重点位置：两侧 / 顶部 / 地面 等]

### 粒子系统
[粒子颜色：冷蓝 / 金色 等]
[粒子行为：随机摆动 / 沿轨迹流动 / 消散 等]
[粒子密度：稀疏 / 密集 等]

### 环境氛围
[整体色调：冷蓝主调 / 暖金主调 等]
[地面效果：反光 / 哑光 等]
[动态衔接：自然流畅 / 无跳变 等]
```

#### Agent 提示词

```
你是科幻空镜头提示词设计大师，能够生成高质量的 6 秒科技类视频空镜头提示词。

## 核心规范

6 秒科技空镜头的标准结构：
1. **镜头运动**（2秒）：镜头沿指定轴线运动，如"沿走廊轴线缓慢向前推进"
2. **光效变化**（2秒）：光源从暗到亮渐变，如"机柜绿色指示灯从暗→亮"
3. **粒子轨迹**（2秒）：粒子沿电流方向流动，如"蓝色电路电流从机柜表面划过"

## 关键要求

- **时长精确**：提示词必须控制在 6 秒内完成
- **动态自然**：强调"柔和无规则"、"无生硬跳变"，避免机械感
- **粒子行为**：用具体数值描述，如"幅度±5%"、"渐变同步"
- **色调统一**：全文色调一致（如冷蓝主调贯穿始终）

## 输出格式

### 镜头设计
[镜头类型 + 持续时间 + 推进幅度]

### 光效描述
[光源类型 + 光效特点 + 重点位置]

### 粒子系统
[粒子颜色 + 粒子行为 + 粒子密度]

### 环境氛围
[整体色调 + 地面效果 + 动态衔接]

## 参考案例

> 6秒科技空镜头，镜头沿走廊轴线缓慢向前推进（6秒内推进1/3画面深度），两侧机柜随机几个绿色指示灯从暗→亮（1秒渐变同步），重点突出：随后两侧各出现一束亮蓝色电路板形电流（带粒子）从机柜表面划过，流向走廊镜头，两侧半透蓝粒子随机小幅度摆动（幅度±5%，柔和无规则），冷蓝主调，地面反光同步电流+粒子轨迹，动态自然衔接，无生硬跳变。
```

## 六、上下文传递机制

### 6.1 ContextContainer 核心设计

ContextContainer 是贯穿整个 Agent 链路的上下文容器，在主 Agent 中创建并维护：

```python
class ContextContainer:
    """贯穿整个 Agent 链路的上下文容器"""
    def __init__(self, user_input: str, image_path: str = None):
        # 用户原始输入
        self.user_input: str = user_input
        self.image_path: Optional[str] = image_path

        # 各 Agent 输出结果
        self.image_description: str = ""    # S1 图片理解结果
        self.draft_prompt: str = ""          # S2 构图生成结果（草稿）
        self.final_prompt: str = ""          # S3 整理优化结果（最终）
        self.variants: list[dict] = []      # S4 风格延展结果（变体列表）

        # 上下文来源
        self.agent_chain: list[str] = []    # 实际走过的 Agent 链路（记录执行顺序）
        self.knowledge_matches: list = []   # 知识库检索结果（K1 关联）
        self.rag_context: list = []         # RAG 检索上下文（Top-3 相似提示词）
        self.user_preference: dict = {}      # 用户偏好（从数据库读取）

        # 状态标记
        self.context_truncated: bool = False  # 是否截断过
        self.last_updated: str = ""         # 最后更新时间
```

### 6.2 完整链路流转规则

```
① 用户提交 → 主Agent 初始化 ContextContainer
                    ↓
② 图片理解？ ──是──→ S1 执行 → image_description 写入 Container
                    ↓（无图片则跳过）
③ RAG 检索（并行） → rag_context 写入 Container
   知识库检索（并行）→ knowledge_matches 写入 Container
                    ↓
④ 构图生成（S2）
   输入：user_input + image_description + rag_context + user_preference
   输出：draft_prompt → Container.draft_prompt
                    ↓
⑤ 整理优化（S3）
   输入：Container.draft_prompt
   输出：final_prompt → Container.final_prompt
                    ↓
⑥ 用户查看结果
   满意 → 收藏 / 结束
   不满意 → 点击"调整"
              ↓
         读取用户选择（Agent + 说明）→ 重新执行指定 Agent
              ↓
         调整结果 → S3 重新整理 → 更新 final_prompt
```

### 6.3 多轮调整时的上下文传递

```
调整轮次 1：
用户选择"风格延展" + 说明"更赛博朋克"
→ Container.final_prompt + user_instruction → S4
→ S4 返回 3 个变体
→ S3 对每个变体逐一整理
→ 展示 3 个整理后变体供用户选择
→ 用户选择变体 B → 覆盖 Container.final_prompt

调整轮次 2：
用户选择"整理优化" + 说明"再短一点"
→ Container.final_prompt + user_instruction → S3
→ S3 重新整理 → 覆盖 Container.final_prompt
→ 展示新结果
```

### 6.4 上下文截断策略

| 字段 | 是否压缩 | 压缩阈值 | 策略 |
|------|---------|---------|------|
| user_input | 否 | — | 始终保留原始值 |
| image_description | 否 | — | 固定简短，不超 500 字 |
| draft_prompt | 是 | >2000 字 | 摘要压缩至 1500 字 |
| final_prompt | 是 | >2000 字 | 摘要压缩至 1500 字 |
| variants | 是 | >3 个 | 保留最新 3 个，早期变体摘要 |
| rag_context | 是 | >3000 字 | 保留 Top-2，丢弃 Top-3 |
| agent_chain | 否 | — | 追加模式，永不截断 |

触发截断时设置 `context_truncated = True`，前端显示提示"上下文已压缩"。

## 八、模型分配策略

### 8.1 核心原则

**混合模式：全局默认模型 + Agent 级覆盖模型**

- 所有 Agent 默认使用全局默认模型（简单场景，一个模型搞定）
- 可为特定 Agent 配置覆盖模型（精细化场景，不同任务用最优模型）
- 覆盖模型的优先级高于全局默认

### 8.4 模型路由逻辑

```python
def get_model_for_agent(agent_id: str) -> dict:
    """获取指定 Agent 应使用的模型"""
    # 1. 查 agent_model_overrides 表
    override = db.get_override(agent_id)
    if override and override.model_id:
        model = db.get_model(override.model_id)
        if model and model.enabled:
            return model

    # 2. 无覆盖或覆盖模型已禁用，返回全局默认
    return db.get_global_default_model()  # priority 最高且 enabled=1 的模型

def db.get_global_default_model() -> dict:
    """获取全局默认模型：priority 最高且 enabled=1"""
    return db.query(
        "SELECT * FROM models WHERE enabled=1 ORDER BY priority DESC LIMIT 1"
    ).fetchone()
```
