### M2：数据层（数据库 + 向量库）

**目标**：完成所有数据表定义及向量库初始化逻辑

**输入**：M1 目录结构
**输出**：可读写的 SQLite 数据库 + 可写入的 FAISS 向量库

**核心内容**：

1. **SQLite 数据库——4 张核心表**：

| 表名 | 字段 | 说明 |  
    |------|------|------| 
   | `prompts` | id, content, category, tags, user_id, created_at, vector_synced | 提示词原文、分类、标签 | 
   | `users` | id, name, style, keywords, tone, default_scene | 用户偏好设置 |
   | `models` | id, vendor, name, api_url, api_key_encrypted, priority, scene | 模型API配置，密钥加密存储 |
   | `logs` | id, timestamp, intent_result, agent_used, model_id, duration_ms | 调度日志 |

2. **FAISS 向量库**：
   - 使用 `faiss-cpu` 的 `IndexFlatIP`（内积）索引
   - 向量维度：m3e-small 输出为 768 维
   - 懒加载：启动时不预加载，收藏提示词时写入
   - 持久化路径：`data/faiss_index/`

**图片存储流程（与 M1/M4/M6 联动）**：

```
前端上传 → POST /api/upload → 后端存 uploads/uuid.ext → 返回路径字符串
主Agent 调用时：image_path = "uploads/uuid.jpg"（相对路径）
VisionAdapter.describe(image_path) → 返回图片文字描述 → 拼入意图识别 prompt
```

> 注意：数据库仅存路径字符串，不存 base64。vision 适配器按需实现，模型选型未定（D-007）。

---

3. **向量嵌入服务**：
   - 模型：`sentence-transformers` 的 `m3e-small`
   - 缓存：首次加载后常驻内存，避免重复初始化
   - CPU优先：`device='cpu'`

**验收标准**：收藏一条提示词后，SQLite 有记录且 FAISS 有对应向量，检索时能返回相似结果。

### M4：主Agent调度层

**目标**：实现意图识别 + 子Agent分发 + 模型路由

**输入**：用户上传的图片+文字需求
**输出**：分发请求至对应专项子Agent

**核心流程**：

1. 接收前端请求（图片路径 + 文字描述）
2. 读取用户偏好（`users` 表）
3. 调用 RAG 检索（获取 Top3 相似提示词作为上下文）
4. **意图识别**：将图片+文字+上下文拼接，通过主Agent prompt 让模型判断场景（电商/海报/其他）
5. 根据识别结果路由至对应子Agent
6. 模型路由：根据子Agent场景标签，从 `models` 表选取同场景最优模型
7. 调用适配层执行

**意图识别 prompt 设计要点**：
- 输入：图片描述 + 文字需求 + 用户偏好 + RAG上下文
- 输出：JSON 格式 `{"scene": "ecommerce|poster|other", "confidence": 0.95, "reason": "..."}`
- 兜底：confidence < 0.7 时返回 `other`，前端引导用户选择分类

**验收标准**：输入"生成立体几何教辅海报"，系统识别为 poster 并调用海报Agent。

### M5：专项子Agent层

**目标**：实现电商Agent 和 海报Agent 的提示词生成逻辑

**输入**：主Agent分发请求 + RAG上下文 + 用户偏好
**输出**：生成的提示词文本

**电商提示词Agent**：

- 核心能力：产品标题、卖点提炼、详情页描述、营销文案
- 输入字段：产品类别、核心卖点、目标人群、关键词埋入要求
- 生成模板：结构化输出（标题/副标题/卖点/详情描述 分段）

**海报设计提示词Agent**：

- 核心能力：构图、色彩、光影、风格、镜头、元素搭配
- 输入字段：参考图片特征、设计风格关键词、场景描述
- 生成模板：分模块输出（主体描述/色彩建议/构图方式/光影氛围）

**验收标准**：两个子Agent 均能独立生成符合场景要求的提示词，RAG 上下文对输出有可感知的改善。

### M7：RAG检索集成

**目标**：将 RAG 检索深度集成至主Agent调度流程

**输入**：用户当前需求
**输出**：Top3 相似提示词作为子Agent上下文

**集成时机**：M7 独立成里程碑，因为涉及 M2（向量库）、M4（主Agent）、M6（前端）的三方联动

**RAG 检索流程**：

1. 用户提交需求 → 主Agent 接收
2. 将需求文本通过 m3e-small 转为向量
3. 在 FAISS 向量库中搜索 Top3（余弦相似度 > 阈值，如 0.5）
4. 取出对应提示词原文，拼入子Agent prompt 作为 Few-shot 示例
5. 子Agent 生成时参考相似提示词风格

**向量相似度阈值**：建议初始设为 0.5，低于该值不返回（避免误导性参考），后续可根据效果调整。

**验收标准**：收藏过"美妆电商标题"后，下次生成类似需求时，RAG 检索能命中并改善生成结果。

## 四、技术方案待确认项

> 以下推荐方案基于 PRD 的约束条件（纯CPU、无显卡、外部API）制定，可直接使用或根据实际情况调整。

### 4.1 API适配器封装方式

**推荐**：采用**类封装**方案（`ModelAdapter` 基类 + 各厂商子类）

```
adapters/
  base.py      → ModelAdapter 基类，定义统一接口 .call(model_id, prompt)
  qwen.py      → 通义千问子类，重写 ._build_request() 和 ._parse_response()
  deepseek.py  → DeepSeek 子类
  minimax.py   → MiniMax 子类
```

**理由**：子类可独立扩展各自厂商的特殊参数处理逻辑，主Agent 调用方只需 `adapter.call(model_id, prompt)`，无需关心底层差异。

**备选**：函数封装（所有逻辑写在一个文件中，通过 vendor 参数分支）——不推荐，后续扩展需反复修改主文件。

### 4.1b 视觉模型适配器（新增）

```
adapters/
  vision/
    base.py      → VisionAdapter 基类，定义统一接口 .describe(image_path)
    qwen_vl.py   → 通义千问VL子类（预留，模型选型未定）
```

**设计要点**：
- `VisionAdapter.describe(image_path)` 接收数据库中的路径字符串，返回图片中文描述
- 当前 vision 模型选型未定，适配器按需实现，不强制绑定任一厂商
- 与文本模型适配器（ModelAdapter）架构平行，互不依赖

### 4.2 嵌入模型的具体使用方式

**推荐**：`sentence-transformers` + `m3e-small`

```python
from sentence_transformers import SentenceTransformer

# 全局单例，启动时加载一次
_model = SentenceTransformer('m3e-small', device='cpu')

def embed_text(text: str) -> list[float]:
    return _model.encode(text, normalize_embeddings=True).tolist()
```

- 初始化约 200MB，CPU 响应 < 1秒，支持中文语义
- `normalize_embeddings=True`：归一化后可直接用余弦相似度（内积）
- 全局单例避免重复加载

### 4.3 FAISS 索引构建时机

**推荐**：懒加载 + 增量索引

```
启动时：
  → 读取 data/faiss_index/ 目录下已有的 .index 文件（如有）
  → 如无则创建新的 IndexFlatIP(dim=768)

收藏时：
  → 将提示词转为向量
  → adapter.add(vector)  # 增量追加
  → adapter.save("data/faiss_index/prompts.index")  # 定期持久化
```

**理由**：

- 避免冷启动时加载全部向量（用户量大了会很慢）
- 与 SQLite 收藏逻辑天然同步
- 定期持久化而非每次收藏都写盘，平衡性能与数据安全

### 4.4 数据库表结构设计

**推荐**：SQLite 共 4 张核心表，关系简洁，扩展平滑

```sql
-- prompts 表：提示词原文
CREATE TABLE prompts (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    content TEXT NOT NULL,          -- 提示词原文
    category TEXT,                  -- 分类（ecommerce/poster/other）
    tags TEXT,                      -- 标签，逗号分隔
    user_id INTEGER DEFAULT 1,      -- 当前版本单用户，固定为1
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    vector_synced INTEGER DEFAULT 0  -- 向量是否已同步至FAISS
);

-- users 表：用户偏好
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name TEXT DEFAULT 'default',
    style TEXT,                     -- 常用风格，如"简约""科技感"
    keywords TEXT,                  -- 常用关键词，逗号分隔
    tone TEXT,                      -- 语气，如"正式""活泼"
    default_scene TEXT              -- 默认场景
);

-- models 表：模型API配置
CREATE TABLE models (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    vendor TEXT NOT NULL,           -- qwen/deepseek/minimax
    name TEXT NOT NULL,             -- 模型名称，如 qwen-turbo
    api_url TEXT NOT NULL,
    api_key_encrypted TEXT NOT NULL, -- AES加密存储
    priority INTEGER DEFAULT 1,    -- 优先级，数字越大越优先
    scene TEXT,                     -- 适用场景，如 ecommerce/poster
    enabled INTEGER DEFAULT 1
);

-- logs 表：调度日志
CREATE TABLE logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    user_id INTEGER DEFAULT 1,      -- 当前版本单用户，固定为1
    intent_result TEXT,             -- 意图识别JSON
    agent_used TEXT,                -- 调用的子Agent名称
    model_id INTEGER,               -- 调用的模型ID
    duration_ms INTEGER,            -- 总耗时（毫秒）
    success INTEGER DEFAULT 1,       -- 是否成功
    error TEXT                      -- 错误信息（如有）
);
```

## 六、风险清单与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| 第三方 API 调用超时/失败率高 | 中 | 高 | 实现自动切换备用模型，记录失败日志 |

| FAISS 向量库损坏/丢失 | 低 | 高 | 收藏时同时写 SQLite 和 FAISS，FAISS 可从 SQLite 重建 |

## 七、开发规范约定

1. **API 响应格式**：统一使用 JSON，包含 `code`（0=成功，-1=失败）、`data`（数据体）、`message`（描述）
2. **错误处理**：后端所有异常捕获并记录到 logs 表，不向用户暴露内部错误堆栈
3. **密钥存储**：`api_key` 入库前使用 `cryptography` 的 Fernet AES 加密
4. **代码风格**：遵循 PEP 8，Python 3.10+ 类型提示（typing）
5. **提交规范**：Git 提交信息格式 `[M1] 完成了xxx`、`[M2] 修复了xxx`
6. **文档维护**：每次里程碑完成后，同步更新 `prd/记忆文档.md` 的进度看板
7. **图片存储规范**：上传图片存 `uploads/`，数据库仅存相对路径字符串，不存 base64
