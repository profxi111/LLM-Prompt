### [D-001] 为什么用 SQLite 而非 PostgreSQL

**日期**：2026-03-27
**决策**：V1.0 采用 SQLite 作为唯一关系型数据库，不引入 PostgreSQL
**理由**：

1. **零配置**：SQLite 为 Python 自带模块（`import sqlite3`），无需安装数据库服务进程
2. **无显卡/低配置友好**：SQLite 纯文件操作，不额外占用内存
3. **单用户场景契合**：V1.0 定位为单机/局域网单用户工具，无并发写入需求
4. **迁移平滑**：SQLite 文件可导出为 CSV，后续如需升级 PostgreSQL 只需改写连接层

**后续影响**：若 V2.0 需要多用户并发支持，需将数据层从 SQLite 迁移至 PostgreSQL，届时 `backend/database/db.py` 的连接管理需要重构。

### [D-002] 为什么选 FAISS-CPU 而非 Chroma

**日期**：2026-03-27
**决策**：RAG 向量库采用 `faiss-cpu` 而非 Chroma
**理由**：

1. **无外部依赖**：Chroma 需要独立服务进程（`chroma run`），增加部署复杂度；FAISS 纯 Python 库，导入即用
2. **CPU 原生支持**：FAISS-CPU 专为 CPU 场景优化，Chroma 默认面向有 Embedding 服务的环境
3. **资源占用低**：Chroma 服务常驻内存，FAISS 轻量很多，适配 4GB 低配设备
4. **与 sentence-transformers 生态契合**：两者均为 Meta（Facebook）出品，原生集成更顺畅

**后续影响**：Chroma 的 metadata 过滤能力比 FAISS 更强，若后续 RAG 场景需要按分类/标签过滤，向量库可能需要升级为 Chroma 或 Qdrant。

### [D-003] 为什么选 m3e-small 作为嵌入模型

**日期**：2026-03-27
**决策**：向量嵌入采用 `sentence-transformers` 的 `m3e-small` 模型
**理由**：

1. **CPU 优先**：m3e-small 专为 CPU 推理优化，初始化约 200MB，推理 < 1秒，无 GPU 依赖
2. **中文语义友好**：m3e 是专为中文语义设计的模型，在中文电商/海报场景下比英文模型效果更好
3. **稳定版推荐**：m3e-small 是稳定版本，m3e-base/m3e-large 可按需升级
4. **输出维度适中**：768 维向量，FAISS IndexFlatIP 检索速度与精度平衡

**备选方案**：

- `text2vec-base-chinese`（也支持中文，维度 768，Ch senteval 评测相近）
- 后续如需英文场景，可考虑 `all-MiniLM-L6-v2`（维度 384，更快）

**后续影响**：当前无定期模型更新计划，嵌入模型更换时需重建 FAISS 索引（从 SQLite prompts.content 重新向量化）。

### [D-004] 为什么用类封装而非函数封装实现模型适配层

**日期**：2026-03-27
**决策**：模型适配层采用 `ModelAdapter` 基类 + 各厂商子类，不使用单一函数 + vendor 参数分支
**理由**：

1. **扩展性**：新增厂商只需新建子类，无需修改主逻辑
2. **封装性**：各厂商的 API 格式差异（header、body、error handling）封装在子类内部
3. **可测试性**：子类可独立 Mock 测试，无需真实 API

**后续影响**：适配器模式已确定，后续新增厂商（Claude、Kimi 等）均遵循此模式。

### [D-005] 为什么用懒加载 + 增量索引而非预加载

**日期**：2026-03-27
**决策**：FAISS 索引采用懒加载（启动不预加载，收藏时增量写入），不预构建完整向量库
**理由**：

1. **冷启动快**：预加载全部向量在用户量增长后耗时严重，影响启动体验
2. **与收藏逻辑同步**：用户收藏 → 生成向量 → 写入 FAISS，流程天然一致
3. **低配友好**：4GB 内存设备预加载大量向量可能导致 OOM

**后续影响**：懒加载意味着首次检索需要等待向量写入；后续若检索延迟高，可考虑定时任务在低峰期批量重建索引。

### [D-006] 为什么图片用文件上传而非 base64 直传

**日期**：2026-03-27
**决策**：前端 multipart/form-data 上传图片至后端，后端存磁盘，数据库存路径字符串
**理由**：

1. **API body 大小限制**：base64 膨胀 33%，易超网关限制；文件上传 body 极小
2. **数据库体积**：路径字符串 vs base64 字符串，差距数十倍
3. **vision API 集成自然**：后端直接读磁盘文件发给视觉模型
4. **前后端职责清晰**：前端只负责传文件，后端负责存储和调用

**后续影响**：所有涉及图片的 API 均传路径字符串（如 `"uploads/uuid.jpg"`），不传 base64。vision 适配器读取磁盘文件。

---

### [D-007] 为什么视觉模型用适配器模式而非硬编码

**日期**：2026-03-27
**决策**：视觉模型（vision API）使用独立适配器层，与文本模型平级，可插拔
**理由**：

1. **vision 模型选型未定**：通义VL / GPT-4o / 其他随时可换，适配器屏蔽差异
2. **与文本模型架构一致**：ModelAdapter（文本）+ VisionAdapter（视觉）双轨并行
3. **扩展性**：新增视觉模型只需新建子类，无需改动主逻辑

**后续影响**：`backend/adapters/vision/` 目录与 `backend/adapters/` 平级，各厂商子类按需实现 `VisionAdapter.describe(image_path)` 接口。

### 问题1：前端按钮无法点击
**日期**：2026-03-27
**问题描述**：
前端界面显示正常，但所有按钮点击都没有反馈，无法进行任何操作。

**原因分析**：
1. HTML中使用了相对路径引用静态文件（`style.css`、`app.js`）
2. FastAPI的静态文件挂载在 `/static/` 路径下（`app.mount("/static", StaticFiles(directory="frontend")`）
3. 浏览器请求相对路径时返回404错误，CSS和JS文件无法加载
4. 没有CSS导致样式异常，没有JS导致事件监听器未绑定

**解决方案**：
修改 `frontend/index.html` 中的静态文件引用路径：
- `<link rel="stylesheet" href="style.css">` → `<link rel="stylesheet" href="/static/style.css">`
- `<script src="app.js"></script>` → `<script src="/static/app.js"></script>`

**验证方法**：
1. 打开浏览器开发者工具（F12）
2. 切换到Console标签页
3. 查看是否有404错误
4. 切换到Network标签页
5. 检查style.css和app.js的状态码是否为200

**防止复发的措施**：
在开发规范中约定：所有静态文件引用必须使用 `/static/` 前缀，禁止使用相对路径。

### 问题2：视觉模型独立服务架构
**日期**：2026-03-27
**问题描述**：
用户需要分别配置文本模型和视觉模型，视觉模型应该作为全局服务，供所有Agent使用，而不是绑定到MasterAgent。

**原因分析**：
1. 原设计将视觉模型集成在MasterAgent中，只服务于意图识别
2. 其他Agent（EcommerceAgent、PosterAgent）无法使用视觉模型
3. 视觉模型职责不清晰，耦合度高

**解决方案**：
采用视觉模型独立服务架构：
1. 创建 `backend/services/vision.py` - VisionService全局单例
2. 创建 `backend/adapters/vision/` 目录，实现视觉模型适配器
3. 修改 MasterAgent，集成 VisionService
4. 新增 `vision_models` 数据库表，独立存储视觉模型配置
5. 新增视觉模型API接口和前端配置界面

**架构优势**：
- ✅ 解耦：视觉模型不绑定到任何Agent
- ✅ 全局可用：任何Agent都可以调用
- ✅ 灵活：可以根据需要选择是否使用视觉模型
- ✅ 易维护：视觉模型配置集中管理
- ✅ 易扩展：新增视觉模型只需修改VisionService

**实现内容**：
- backend/adapters/vision/base.py - VisionAdapter基类
- backend/adapters/vision/qwen_vl.py - 通义千问VL适配器
- backend/services/vision.py - VisionService全局服务
- backend/database/migrations.py - 新增vision_models表
- backend/main.py - 新增视觉模型API接口
- backend/agents/master.py - 集成VisionService
- frontend/index.html - 新增视觉模型配置界面
- frontend/app.js - 新增视觉模型配置功能

**防止复发的措施**：
视觉模型采用独立服务架构，与文本模型平级，职责清晰。

## 六、运维操作指南

> 记录服务的启动、停止、性能监控等运维操作，便于日常开发和部署使用。

### 6.1 启动服务

**Windows PowerShell 环境**：
```powershell
# 设置Python路径（解决模块导入问题）
$env:PYTHONPATH="d:\trae test"

# 启动服务
python backend/main.py
```

**或使用 uvicorn**（推荐）：
```powershell
$env:PYTHONPATH="d:\trae test"
python -m uvicorn backend.main:app --host 0.0.0.0 --port 8000
```

**服务信息**：
- 默认端口：8000
- 访问地址：http://localhost:8000/
- 局域网访问：http://[本机IP]:8000/

### 6.2 停止服务

**方法1：Ctrl+C 终止**
- 在运行服务的终端窗口按 `Ctrl + C`

**方法2：查找并终止进程**
```powershell
# 查找占用8000端口的进程
netstat -ano | findstr :8000

# 终止指定PID的进程（替换21912为实际PID）
taskkill /F /PID 21912
```

**方法3：终止所有Python进程**
```powershell
Get-Process -Name python -ErrorAction SilentlyContinue | Stop-Process -Force
```

### 6.3 性能监控

**查看进程资源占用**：
```powershell
# 查看Python进程的CPU、内存、线程数
Get-Process -Name python | Select-Object Id, ProcessName, CPU, @{Name='内存(MB)';Expression={[math]::Round($_.WorkingSet/1MB,2)}}, @{Name='线程数';Expression={$_.Threads.Count}}
```

**查看网络连接**：
```powershell
# 查看8000端口的活跃连接
netstat -ano | findstr :8000 | findstr ESTABLISHED
```

**性能基准参考**（V1.0版本）：
- CPU使用率：3%左右（空闲状态）
- 内存使用：90MB左右
- 线程数：33个
- 活跃连接：根据用户数量动态变化

### 6.4 常见问题排查

**问题1：ModuleNotFoundError: No module named 'backend'**

**问题2：[Errno 10048] 端口已被占用**
- **原因**：8000端口已被其他进程占用
- **解决**：查找并终止占用进程（见6.2节）

**问题3：前端按钮无法点击**
- **原因**：静态文件路径配置错误（404错误）
- **解决**：确保HTML中使用 `/static/` 前缀引用CSS和JS文件

**问题4：前端样式异常**
- **原因**：CSS文件未加载
- **解决**：检查浏览器控制台是否有404错误，确认路径为 `/static/style.css`

**核心依赖**：
- fastapi - Web框架
- uvicorn - ASGI服务器
- faiss-cpu - 向量检索（CPU版）
- sentence-transformers - 文本嵌入（可选，未安装时使用简单哈希）
- cryptography - API密钥加密
- requests - HTTP请求
- python-multipart - 文件上传支持
